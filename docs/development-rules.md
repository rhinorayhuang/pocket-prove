# 開發規範

## 不可違反的規則

1. 所有新功能都必須保持 `npm run build` 可通過。
2. 所有中文檔案與程式碼檔案都使用 UTF-8。
3. 不要把遊戲規則寫進 Vue 展示元件。
4. 不要把內容資料硬編碼在多個地方。
5. 修改結構時，必須同步更新文件。

## 提交前最低要求

每次完成一個功能前，至少要做：

1. 確認型別沒有壞掉
2. 執行 `npm run build`
3. 確認 `dist/` 可重新輸出
4. 如果有新增模組，更新相關文件

## 實作原則

### 資料驅動優先

新增怪獸、地圖、任務、技能時：
1. 優先放進 `src/game/data/`
2. 用 `types/` 描述結構
3. 用 `systems/` 或 `scenes/` 消費資料

### 單一責任

1. 一個檔案只解決一類問題
2. 一個 system 只負責一類規則
3. 一個 component 只處理一種畫面責任

### 純函式優先

在 `systems/` 中：
1. 優先寫純函式
2. 盡量減少直接修改外部 state
3. 若要改 state，讓呼叫端明確知道

## 禁止事項

1. 不要直接在 `App.vue` 裡寫戰鬥傷害計算
2. 不要在 `renderer.ts` 裡塞劇情分支
3. 不要在 `types/` 放執行邏輯
4. 不要在 `data/` 放副作用程式
5. 不要用臨時字串到處判斷 quest 狀態

## 編碼與顯示

PowerShell 可能把中文顯示成亂碼。

這代表：
1. 不要只靠終端機輸出判斷檔案是否壞掉
2. 優先以編輯器、瀏覽器、build 結果判斷
3. 文件內容若需要驗證，應從實際檔案或 UI 確認

## 何時一定要補文件

以下變更必須同步更新文件：

1. 新增資料夾層級
2. 新增場景種類
3. 新增系統模組
4. 改變 quest 流程
5. 改變 battle action 規格
6. 改變 save data 結構

## 建議工作流程

1. 先改 `types`
2. 再改 `data`
3. 再改 `systems`
4. 再改 `scenes`
5. 最後接到 `engine` 和 `components`

這個順序最容易避免邏輯分散和型別失控。
